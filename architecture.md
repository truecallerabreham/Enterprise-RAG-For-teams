# Enterprise RAG Architecture

This document provides a deep-dive, end-to-end breakdown of the Enterprise RAG Architecture. The system is designed to handle multi-repository, large-scale codebases (2M+ LOC) with high accuracy, security, and incremental freshness.

---

## Phase 1: The Ingestion Pipeline (How the AI Reads the Code)
This phase is triggered completely asynchronously whenever code is pushed to your repositories. It does not wait for a user to ask a question.

### 1. Git Webhook & Ingestion Walker
When a developer pushes a commit, a webhook fires. The "Ingestion Walker" does not re-read the whole 2M-LOC codebase; it only looks at the `git diff` to see exactly which files changed. 

### 2. Tree-sitter Parser & Logical Boundaries
Instead of splitting text arbitrarily (e.g., every 500 words), we use **Tree-sitter**, which reads the code like a compiler. It builds an Abstract Syntax Tree (AST) and perfectly slices the code at logical boundaries: taking exactly one function, one class, or one method per "chunk."

### 3. AST Hashing (The Update Trigger)
Before expensive processing, we generate a mathematical hash of the chunk's AST. 
*   **No Change:** If the developer only fixed a typo in a comment, the AST hash remains identical. We skip processing.
*   **Logic Change:** If the actual logic changed, the hash changes, and we send the chunk down the pipeline.

### 4. The Three-Pronged Representation
Each chunk is represented in three ways to ensure maximum discoverability:
*   **Dense Embeddings (Voyage-code-3):** Handles semantic searches (e.g., "user login" matches `authenticate_client`).
*   **Sparse Keyword Index (Tantivy/BM25):** Handles exact matches (e.g., specific variable names like `MAX_RETRY_COUNT_V2`).
*   **AI Summary (Claude Haiku 4.5):** Generates a 1-sentence summary of the "public contract."

### 5. Symbol Resolver & Inverted Index
We look at every `import` statement and log relationships into a hardcoded **Inverted Index**. This maps cross-repo dependencies so the AI knows how the repositories interact without guessing.

---

## Phase 2: The Storage Layer (Where the Data Lives)

### 1. Qdrant (The Vector Database)
Holds Dense Vectors, the Sparse BM25 index, raw code text, and summaries. Crucially, it handles **Metadata-level RBAC**. Every chunk is tagged with an `allowed_groups` array (from SSO/Active Directory) to ensure users only see code they are authorized to access.

### 2. Kuzu or Neo4j (The Graph Database)
Maps the structural relationships. If `PaymentService` calls `AuthService`, a physical edge exists here to facilitate cross-repo retrieval expansion.

---

## Phase 3: The Query Pipeline (How the AI Answers Questions)
The query workflow is orchestrated using **LangGraph**.

### 1. Auth Service & Pre-Filtering
The system checks the user's SSO token and retrieves their authorized groups before the search begins.

### 2. The Retrieve Node & RRF
*   **RBAC Filtering:** Qdrant drops unauthorized code chunks immediately.
*   **Hybrid Search:** Runs Dense and BM25 search simultaneously.
*   **Reciprocal Rank Fusion (RRF):** Mathematically merges rankings to ensure the most relevant chunks float to the top regardless of scoring method.

### 3. Cross-Repo Expansion
The system uses the Graph DB to pull in connected code (e.g., utility functions in different repos) that the top retrieved chunks depend on.

### 4. The Reranker Node (Cross-Encoder via ONNX/TensorRT)
The top 30-50 chunks are re-scored by a heavy Cross-Encoder model. Optimized with **ONNX/TensorRT**, this model accurately ranks relevance out of 100, filtering out the bottom matches.

### 5. The Synthesis Node (Claude Sonnet 4.7)
The final answer is generated by Claude Sonnet 4.7 using the top 10 reranked chunks. It is instructed to cite sources (Repo, File, Line) using strict JSON formatting.

### 6. The Citation Validation Loop
The system intercepts the JSON citations and verifies they exist in the provided context.
*   **Verified:** The answer is shown to the user.
*   **Hallucination Detected:** LangGraph loops back to the LLM to fix the citation using only the available context (max 2 retries).

---

## Why this is "Enterprise Grade"
1.  **Security:** Solved by Qdrant RBAC pre-filtering.
2.  **Accuracy:** Hallucinations are caught by the Citation Validation Loop.
3.  **Scale:** Rerankers handle the noise of large codebases.
4.  **Efficiency:** AST Hashing prevents unnecessary compute on minor changes.
